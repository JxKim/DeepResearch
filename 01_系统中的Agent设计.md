# AI 研究报告工作台 Agent 设计

## 1. 设计目标

本系统的 Agent 设计服务于一个核心目标：

> 帮助企业内部分析师从模糊研究问题出发，经过研究设定、资料收集、事实整理、洞察形成和报告撰写，产出可追溯、可编辑、可复用的研究报告。

第一版不追求复杂的 Agent 数量，但需要保留研究产品中最核心的三个职责。

因此，MVP 阶段实现 3 个智能体：

1. 研究管理智能体
2. 信息检索智能体
3. 报告生成智能体

其中，研究管理智能体是 DeepAgents 的主智能体，负责整体协调和决策。

信息检索智能体和报告生成智能体作为研究管理智能体可调用的子智能体或工具存在，不单独控制完整流程。

## 2. 为什么第一版是三个智能体

研究报告产品中有三类能力差异很明显：

- 研究管理：理解问题、拆解大纲、判断研究路径、决定何时调用其他能力。
- 信息检索：围绕具体问题查找资料、筛选来源、整理证据链。
- 报告生成：把研究结果转化为结构清晰、带引用、带图表的 HTML 报告。

这三类能力的目标、输入、输出和质量标准都不同。

如果全部放在一个智能体里，主智能体的 Prompt 会变得过长，职责也会变得不清晰。

如果拆得更多，第一版又会增加调试成本。

所以第一版使用三个智能体是合适的折中：

- 数量足够少，便于调试。
- 职责足够清楚，便于实现。
- 符合 DeepAgents 中“主智能体协调子智能体和工具”的使用方式。

## 3. 三个智能体的关系

```text
用户
  ↓
研究管理智能体
  ├── 调用信息检索智能体
  ├── 调用报告生成智能体
  └── 调用普通工具
```

研究管理智能体是唯一直接面向用户研究目标的智能体。

信息检索智能体只接收研究管理智能体分派的原子检索问题。

报告生成智能体只接收研究管理智能体整理后的大纲、事实、洞察、图表数据和来源列表。

## 4. 研究管理智能体

研究管理智能体是系统主智能体。

它负责完整研究过程的协调，但不直接承担所有底层能力。

职责包括：

- 理解用户输入的研究主题。
- 澄清研究目标、研究范围、目标读者和默认假设。
- 生成研究任务书。
- 生成研究大纲，并允许用户修改或替换。
- 根据大纲拆解需要回答的研究问题。
- 判断哪些问题需要外部公开资料。
- 判断哪些问题需要内部知识库。
- 将原子检索问题交给信息检索智能体。
- 汇总信息检索智能体返回的来源、事实和证据链。
- 标记信息冲突、不确定性和引用来源。
- 基于事实形成洞察。
- 将大纲、事实、洞察、图表数据和来源交给报告生成智能体。
- 保存研究过程中的结构化结果，便于后续版本管理和复用。

研究管理智能体不负责：

- 逐个网页抓取正文。
- 直接处理 RAGFlow 底层检索细节。
- 直接生成最终 HTML 页面样式。
- 直接管理数据库连接。
- 直接调度 Celery 长任务。

## 5. 信息检索智能体

信息检索智能体负责资料检索和证据整理。

它接收研究管理智能体分派的原子问题，例如：

- 某行业未来三年的市场规模是多少？
- 主要竞争玩家有哪些？
- 某项政策何时发布，影响是什么？
- 公司内部是否有类似项目经验？

职责包括：

- 根据原子问题生成检索关键词。
- 调用外部搜索工具检索公开资料。
- 调用网页读取工具提取正文、发布时间、作者和来源。
- 按需调用 RAGFlow 工具检索内部知识库。
- 筛选低质量来源。
- 整理事实卡片。
- 整理来源列表。
- 标记数据口径、时间范围和冲突信息。
- 返回可被研究管理智能体使用的证据链。

信息检索智能体不负责：

- 决定整份报告的大纲。
- 生成最终研究结论。
- 编写完整报告。
- 和用户直接交互。

## 6. 报告生成智能体

报告生成智能体负责将研究结果转化为最终报告。

它接收研究管理智能体传入的材料：

- 已确认大纲
- 事实卡片
- 洞察卡片
- 来源列表
- 图表数据
- 报告风格要求

职责包括：

- 生成 HTML 报告结构。
- 组织摘要、核心结论、正文、风险和建议。
- 将关键事实和判断绑定引用。
- 生成图表区域和图标说明。
- 生成参考来源列表。
- 保证报告表达适合企业内部阅读。

报告生成智能体不负责：

- 自己重新做大规模搜索。
- 自己决定是否调用内部知识库。
- 修改研究项目状态。
- 和用户直接交互。

## 7. ReAct 执行方式

三个智能体都可以基于 ReAct 思想工作，但职责范围不同。

研究管理智能体的 ReAct：

- 判断研究下一步应该做什么。
- 决定是否需要调用信息检索智能体。
- 决定是否需要调用报告生成智能体。
- 判断证据是否足够支撑报告生成。

信息检索智能体的 ReAct：

- 判断应该使用哪些搜索关键词。
- 判断是否需要读取网页详情。
- 判断是否需要补充内部知识库。
- 判断来源之间是否存在冲突。

报告生成智能体的 ReAct：

- 判断报告结构是否完整。
- 判断关键结论是否有引用支撑。
- 判断图表和图标是否服务于报告表达。
- 判断哪些内容需要标注不确定性。

ReAct 的目标不是让智能体无限自由探索，而是让每个智能体在自己的职责边界内完成任务。

## 8. 用户介入点

第一版只保留少量关键人工介入点。

用户需要介入的地方：

- 研究设定阶段：用户可以补充研究目标、范围、读者和是否使用内部知识库。
- 大纲阶段：用户可以采用、修改或替换系统生成的大纲。
- 报告阶段：用户可以修改结论、章节、表达方式，并通过引用核查关键事实。

用户不需要介入的地方：

- 不需要逐条确认事实卡片。
- 不需要在资料收集中间判断每个事实是否可信。
- 不需要手动组织证据链。

中间过程可以允许用户查看来源列表、排除明显不合适来源、补充指定资料，但这不是必经步骤。

## 9. 第一版不做的事情

第一版不做以下事情：

- 不拆超过三个智能体。
- 不引入独立工作流引擎。
- 不让信息检索智能体和报告生成智能体直接面向用户。
- 不让多个智能体互相循环调用。
- 不在中间流程要求用户逐条确认事实。

## 10. 未来演进

未来只有在任务复杂度明显上升时，才考虑新增智能体。

可能的演进方向：

- 数据分析智能体：专门处理市场规模估算、表格分析和财务数据。
- 图表智能体：专门生成更复杂的图表和可视化组件。
- 质量审核智能体：专门检查引用完整性、逻辑矛盾和报告质量。

这些都不是第一版必须实现的内容。
