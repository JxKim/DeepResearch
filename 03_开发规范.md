# 项目开发规范

## 1. 关于编码前置要求

每次开始编码前，必须先阅读本文件，确认本次修改符合当前项目规范。

如果需求和本规范冲突，优先停止编码并说明冲突点，不要直接绕过规范实现。

## 2. 关于注释

所有类、核心方法和核心函数都必须有中文注释，说明该类或方法的作用、输入输出和职责边界。

注释应该解释业务意图和设计边界，不写无意义注释。

## 3. 关于代码设计

不允许过度复杂设计，不需要过度抽象。不要自作聪明，使用一些无用的内容

只有在减少重复代码、统一稳定接口或隔离明确外部依赖时，才允许增加抽象。

代码要做到简洁、清晰、一目了然。第一版优先完成可运行的 MVP，不追求一次性设计完整平台。

## 4. 关于 Agent 设计

第一版实现三个智能体。

三个智能体分别是研究管理智能体、信息检索智能体、报告生成智能体。

研究管理智能体是 DeepAgents 主智能体，负责协调信息检索智能体、报告生成智能体和普通工具。

三个智能体都可以基于 ReAct 思想执行任务，但必须遵守各自职责边界。

第一版不要拆超过三个 Agent，不要引入独立工作流引擎。

如果后续需要新增智能体，必须先说明拆分原因、职责边界和收益。

## 5. 关于代码调试

核心方法调用过程、核心参数生成、Agent 工具调用、外部服务调用和长任务状态变化，必须记录日志。

日志统一使用 loguru 或标准 logger 模块实现。

日志级别要求：

- `logger.info`：记录关键业务流程和任务状态变化。
- `logger.debug`：记录调试所需的中间参数和模型输入输出摘要。
- `logger.warning`：记录可恢复但需要关注的问题。
- `logger.error`：记录不可忽略的异常和失败结果。

日志中不得输出 API Key、访问令牌、企业内部敏感原文和用户隐私信息。

## 6. 关于防御性编程

减少无意义的防御性编程。

不要在 `try` 中动态 `import` 包。

不要用宽泛的 `try except` 吞掉异常。

只有和业务相关的条件未满足，或外部服务调用可能失败时，才需要捕获并处理异常。

接口入参可以依赖 Pydantic 做结构化校验，不需要在每个接口内部重复手写空值校验。

## 7. 关于调用 LLM

调用 LLM 时，优先使用 LangChain 的 `ChatOpenAI`、`ChatDeepSeek` 或同类 Chat Model 封装。

需要结构化输出时，必须使用 `with_structured_output` 或等价的结构化输出方式。

Prompt 必须维护到外部配置中，不允许散落硬编码在业务代码里。

模型名称、温度、API 地址、API Key、搜索服务地址、RAGFlow 地址等配置不得硬编码，必须通过配置文件或环境变量管理。

## 8. 关于静态类型标注

所有函数参数、返回值、核心变量都需要静态类型标注。

示例：

```python
message_list: list[str]
```

优先使用 Python 3.12 原生类型写法，例如 `list[str]`、`dict[str, Any]`。

## 9. 关于每次生成的代码数量

一次仅能生成 2-3 个文件的代码。

每个文件代码量不得超过 500 行。

项目总代码量第一版不得超过 3000 行，不包含 Prompt、配置、文档和测试数据。

不得在代码中写硬编码业务常量。稳定枚举、默认值和 Prompt 路径应放到配置或常量模块中。

## 10. 关于测试和验证

每次完成核心功能后，必须至少做一次可运行验证。

优先验证：

- 接口能否启动。
- 核心函数能否被调用。
- Agent 工具输入输出结构是否正确。
- 报告生成结果是否包含引用和来源列表。

如果因为缺少外部依赖、API Key 或服务未启动导致无法验证，必须在最终说明中明确写出。

## 11. 关于提交 commit

commit 一律使用中文描述。

提交前需要确认：

- 没有提交 `.env`、API Key、访问令牌等敏感信息。
- 没有提交无关临时文件。
- 没有把用户未要求修改的内容混入提交。
