# AI 研究报告工作台后端技术栈设计

## 1. 设计目标

本系统后端服务的核心目标是支撑“研究项目”从创建到报告交付的完整流程。

它需要支持：

- 长周期研究任务
- 三个智能体协作执行
- 人工介入和暂停恢复
- 外部信息检索
- 内部知识库检索
- 事实卡片和洞察卡片管理
- 报告草稿生成和版本管理

本设计明确不使用 PostgreSQL。

## 2. 总体技术选型

### 2.1 后端语言

推荐使用 Python 作为主要后端语言。

原因：

- LangChain、DeepAgents、LangGraph 生态以 Python 支持最成熟。
- 研究、检索、文档解析、数据处理、Agent 编排等能力更容易整合。
- 后续接入大模型、搜索、知识库、PDF 解析、Office 文档处理时，Python 工具链更完整。

### 2.2 Web 框架

只能使用FastAPI

FastAPI 适合本系统的原因：

- 原生支持异步接口。
- 适合构建 API 服务。
- 和 Pydantic 结合紧密，便于定义结构化数据。
- 适合处理前端工作台需要的流式状态更新、任务查询和后台任务触发。

### 2.3 Agent 执行

使用：

- LangChain DeepAgents

DeepAgents 用于构建具备规划、子智能体调用、工具调用和上下文文件系统能力的研究型智能体。

第一版不引入独立工作流引擎，不把研究流程拆成固定节点状态机。

第一版的执行方式是：

- 研究管理智能体作为 DeepAgents 主智能体，负责理解研究任务、规划研究路径和协调执行。
- 信息检索智能体负责外部搜索、网页读取、RAGFlow 检索、事实整理和证据链返回。
- 报告生成智能体负责将大纲、事实、洞察、图表数据和来源列表转化为 HTML 报告。
- 三个智能体都可以基于 ReAct 思想工作，但职责边界不同。
- FastAPI 负责项目接口、状态查询和结果保存。
- 第一版先使用 FastAPI 的 `asyncio.create_task` 承载后台任务。

LangGraph 可以作为 DeepAgents 底层依赖或未来扩展能力，但第一版不把它作为独立的产品流程编排层。

### 2.4 主数据库

推荐使用 MongoDB。

原因：

- 研究项目、报告草稿、事实卡片、洞察卡片、用户反馈等数据结构存在较强的文档属性。
- 研究过程中的中间状态变化频繁，字段结构可能不断演进。
- 适合保存嵌套结构，例如章节、引用、证据、版本、反馈记录。
- 不使用 PostgreSQL 的前提下，MongoDB 是比较适合该类产品的主业务库。

MongoDB 主要保存：

- 用户
- 研究项目
- 研究任务书
- 研究大纲
- 信息源
- 事实卡片
- 洞察卡片
- 报告草稿
- 报告版本

### 2.5 缓存与任务状态

推荐使用 Redis。

Redis 主要用于：

- 短期任务状态缓存
- WebSocket / SSE 推送状态
- 分布式锁
- 限流
- 临时上下文
- 任务队列消息中转
- 用户会话缓存

Redis 不作为长期业务数据存储。

### 2.6 知识库与 RAG 引擎

推荐优先评估 RAGFlow 作为内部知识库检索和文档理解引擎。

RAGFlow 是一个开源 RAG 引擎，特点是围绕复杂文档理解、知识库构建、可追溯引用和企业级 RAG 工作流展开。它适合处理 PDF、Word、Excel、PPT、图片、扫描件、网页等复杂资料，并提供知识库、文件管理、检索和 API 能力。

在本系统中，RAGFlow 不建议作为完整产品底座，而建议作为 Knowledge Service 的底层能力。

它主要负责：

- 内部文档上传
- 文档解析
- 复杂版面理解
- 文档切分
- 知识库管理
- 语义检索
- 引用片段返回
- 内部资料问答
- 知识库 API 调用

本系统自身仍然负责：

- 研究项目管理
- 研究任务状态管理
- 三个智能体执行
- 人工介入
- 事实卡片
- 洞察卡片
- 报告生成
- 用户反馈沉淀
- 权限和业务规则

也就是说，RAGFlow 是知识库引擎，不是研究报告工作台本身。

### 2.7 对象存储

推荐使用 MinIO 或云厂商对象存储。

对象存储用于保存：

- 用户上传文件
- 内部知识库原始文件
- PDF
- Word
- Excel
- PPT
- 图片
- 生成的报告文件
- 图表文件
- 中间解析产物

如果是内部自用部署，MinIO 是合适的默认选择。

### 2.8 搜索引擎

推荐使用 Tavily。

Tavily 用于：

- 全文搜索
- 文档标题搜索
- 报告内容搜索
- 来源搜索
- 内部资料关键词搜索
- 研究项目搜索

### 2.9 后台任务执行

第一版不强制引入独立任务队列。

第一版采用：

- FastAPI 接收请求。
- MongoDB 保存项目状态和任务状态。
- `asyncio.create_task` 在 API 进程内启动后台任务。

适用任务：

- 研究任务书和大纲生成。
- 报告生成。
- 轻量级资料整理。

这样做的原因是第一版优先验证产品链路，不先引入 Celery、Redis 和独立 Worker 的复杂度。

第一版需要接受的限制：

- API 进程重启时，正在执行的后台任务可能中断。
- 不适合大量并发长任务。
- 不适合多 API worker 同时承载后台任务。
- 失败重试需要后续单独补充。

当出现以下情况时，再引入 Celery + Redis 或 Dramatiq + Redis：

- 多用户并发研究任务明显增加。
- 需要独立扩容后台 Worker。
- 需要成熟的失败重试和任务超时控制。
- 需要任务在 API 服务重启后继续执行。
- 需要对不同任务类型做并发控制和资源隔离。

也就是说，第一版先用 API 进程内后台任务，生产化阶段再引入独立任务队列。

## 3. 第一版最小可行后端

第一版一定需要的功能：

- 结合用户问题，生成研究设定、研究大纲和任务计划
- 整理互联网搜索和 RAGFlow 搜索，形成逻辑链条和研究洞察。在报告中，相关数据必须可溯源，报告末尾必须有参考目录
- 至少要有一个明确的人工介入点，第一版优先放在大纲确认阶段
- 最终需要生成带图表的报告，可以先生成 HTML 版本，不需要先做 PDF 版本
- 一定要是可以在生产环境下，通过容器化的方式扩容部署的

第一版不需要的功能：
- 权限相关
- 登录验证相关
