# DeepResearch 生产化改造方案

本文档用于说明当前 DeepResearch 项目如何从 MVP 演进到更接近生产级项目，重点补强两块能力：

- 生产级日志追踪，尤其是 LLM / Agent 调用链路追踪
- 使用 Celery + Redis 承载长耗时后台任务

当前项目已经具备清晰的主流程：

```text
创建研究项目 -> 生成研究任务书和大纲 -> 人工确认大纲 -> 执行研究 -> 生成 HTML 报告
```

但如果要在简历或面试中称为“生产级项目”，需要进一步补充可观测性、任务可靠性、失败恢复和部署运行能力。

## 1. 为什么需要生产化改造

当前 MVP 版本的后台任务主要通过 `asyncio.create_task` 在 API 进程内执行。

这种方式适合教学演示和本地验证，优点是简单、依赖少、容易理解。但它也有明显限制：

- API 进程重启后，正在执行的任务会中断
- 多个 API worker 场景下，任务调度不可控
- 不方便限制不同任务类型的并发
- 不方便做失败重试、任务超时、任务取消
- 缺少统一的任务执行日志和链路追踪

AI 研究报告类任务通常属于长耗时任务，可能包含多轮 LLM 调用、外部搜索、网页读取、RAG 检索和报告生成。因此生产化阶段应该把任务执行从 API 进程中拆出去，由专门的 Worker 承载。

## 2. 生产级日志追踪目标

生产级日志不是简单地多打几行 `logger.info`，而是要能回答以下问题：

- 某个用户请求触发了哪个研究项目
- 某个研究项目创建了哪些后台任务
- 某个后台任务执行了哪些 Agent 步骤
- 每次 LLM 调用的耗时、模型、输入输出摘要是什么
- 每次工具调用的耗时、参数摘要、结果数量、失败原因是什么
- 任务失败时，失败发生在哪个阶段
- 同一个 `trace_id` 下能否串起 HTTP 请求、Celery 任务、Agent 调用和工具调用

建议建立以下追踪字段：

| 字段 | 含义 |
| --- | --- |
| `trace_id` | 一次完整业务链路的追踪编号 |
| `request_id` | 单次 HTTP 请求编号 |
| `project_id` | 研究项目编号 |
| `task_id` | 后台任务编号 |
| `celery_task_id` | Celery 自身任务编号 |
| `agent_name` | Agent 名称，例如 research-manager-agent |
| `step_name` | 当前执行阶段，例如 generate_report |
| `tool_name` | 工具名称，例如 external_search |
| `model_name` | LLM 模型名称 |
| `duration_ms` | 调用耗时 |
| `status` | 执行状态，例如 ok / skipped / error |
| `error_type` | 异常类型 |

## 3. 日志格式建议

建议使用结构化 JSON 日志，便于后续接入 ELK、Loki、Datadog、阿里云日志服务或 OpenTelemetry Collector。

示例：

```json
{
  "time": "2026-06-07T12:00:00Z",
  "level": "INFO",
  "event": "agent.invoke.finished",
  "trace_id": "7f1c9c5d8b9b4d6a",
  "project_id": "project-001",
  "task_id": "task-001",
  "agent_name": "research-manager-agent",
  "step_name": "generate_report",
  "model_name": "openai:gpt-4.1-mini",
  "duration_ms": 18432,
  "status": "ok"
}
```

工具调用日志示例：

```json
{
  "time": "2026-06-07T12:00:03Z",
  "level": "INFO",
  "event": "tool.call.finished",
  "trace_id": "7f1c9c5d8b9b4d6a",
  "project_id": "project-001",
  "task_id": "task-001",
  "tool_name": "external_search",
  "query_hash": "sha256:xxxx",
  "result_count": 5,
  "duration_ms": 1260,
  "status": "ok"
}
```

注意：日志中不应该直接打印完整 Prompt、API Key、用户敏感信息或大段网页正文。生产日志应该保存摘要、长度、hash、数量和错误类型。

## 4. LLM 调用链路追踪设计

LLM 调用链路可以按三层记录。

第一层：业务任务级别。

记录一次研究任务从开始到结束的整体状态，例如：

- `research_brief.generate.started`
- `research_brief.generate.finished`
- `outline.revise.started`
- `report.generate.started`
- `report.generate.failed`

第二层：Agent 调用级别。

记录研究管理 Agent、检索 Agent、报告生成 Agent 的调用情况，例如：

- Agent 名称
- 任务类型
- 模型名称
- 输入摘要长度
- 输出摘要长度
- 耗时
- 是否成功

第三层：工具调用级别。

记录外部搜索、RAGFlow 检索、网页读取、HTML 报告生成等工具调用，例如：

- 工具名
- 参数摘要
- 返回结果数量
- 失败类型
- 耗时

这样面试时可以解释为：

> 我没有只关注 LLM 最终返回结果，而是把一次报告生成拆成任务、Agent、工具三层追踪。这样当报告生成慢、失败或引用来源异常时，可以定位到底是模型调用慢、外部搜索失败、RAGFlow 无结果，还是报告生成阶段解析失败。

## 6. Celery 任务后端设计

建议在生产化阶段引入：

```text
Celery + Redis
```

Redis 角色：

- Celery broker
- Celery result backend
- 后续可扩展为任务锁、限流、短期状态缓存

Celery Worker 负责执行长耗时任务：

- 生成研究任务书和大纲
- 修改研究大纲
- 生成研究报告

API 服务只负责：

- 接收请求
- 创建项目和任务记录
- 把任务投递给 Celery
- 返回 `task_id`
- 提供任务状态查询接口

Worker 服务负责：

- 读取项目数据
- 调用 Agent
- 调用工具
- 保存结果
- 更新任务状态
- 记录执行日志

## 7. Celery 文件结构建议

建议新增：

```text
app/task_queue/
  __init__.py
  celery_app.py
  celery_tasks.py
```

`celery_app.py` 负责创建 Celery 实例：

```python
from celery import Celery

from app.config.config import get_settings

settings = get_settings()

celery_app = Celery(
    "deep_research",
    broker=settings.celery_broker_url,
    backend=settings.celery_result_backend,
)

celery_app.conf.update(
    task_serializer="json",
    result_serializer="json",
    accept_content=["json"],
    timezone="UTC",
    task_track_started=True,
    worker_prefetch_multiplier=1,
)
```

`celery_tasks.py` 负责定义任务：

```python
@celery_app.task(name="research.generate_research_brief", bind=True, max_retries=2)
def generate_research_brief_task(self, project_id: str, task_id: str, trace_id: str) -> None:
    ...

@celery_app.task(name="research.revise_outline", bind=True, max_retries=2)
def revise_outline_task(
    self,
    project_id: str,
    task_id: str,
    revision_instruction: str,
    trace_id: str,
) -> None:
    ...

@celery_app.task(name="research.generate_report", bind=True, max_retries=1)
def generate_report_task(
    self,
    project_id: str,
    task_id: str,
    user_instruction: str | None,
    trace_id: str,
) -> None:
    ...
```

由于当前项目大部分执行逻辑是 async 函数，Celery 任务中可以通过 `asyncio.run(...)` 调用已有异步执行流程。

## 8. 任务后端兼容策略

建议不要直接删除当前 `asyncio.create_task` 实现，而是增加一个配置项：

```text
TASK_BACKEND=asyncio
TASK_BACKEND=celery
```

本地教学演示：

```text
TASK_BACKEND=asyncio
```

生产或准生产环境：

```text
TASK_BACKEND=celery
```

这样可以在简历和面试中说明：

> 系统支持本地轻量模式和生产任务队列模式。本地开发使用 asyncio 降低依赖，生产环境切换到 Celery + Redis，实现任务解耦、失败重试、超时控制和 Worker 横向扩展。

## 10. 失败重试策略

不同任务不应该使用完全相同的重试策略。

建议：

| 任务类型 | 是否重试 | 原因 |
| --- | --- | --- |
| 生成研究任务书和大纲 | 可重试 2 次 | 通常是 LLM 或临时网络失败 |
| 修改大纲 | 可重试 2 次 | 输入较短，重试成本低 |
| 生成报告 | 可重试 1 次 | 成本较高，避免重复消耗过多 Token |
| 外部搜索 | 工具内部可短重试 | 常见网络抖动 |
| 网页读取 | 不强制重试 | 很多失败来自站点限制 |
| RAGFlow 检索 | 可重试 1 次 | 内部服务可能短暂不可用 |

失败后应该更新 MongoDB 中的任务状态：

```text
running -> failed
```

失败信息只保存摘要，例如：

```text
报告生成失败: TimeoutError
```

详细异常堆栈进入日志系统，不直接暴露给前端。

## 13. Agent 运行过程可视化增强

除了结构化日志和 Celery 任务后端，还建议增加一套面向前端的实时事件推送机制。

日志系统主要服务开发、运维和问题排查；实时事件推送主要服务用户体验和教学演示。两者关注点不同，不应该混在一起。

建议增加：

```text
WebSocket 或 SSE 任务事件流
```

用于实时展示：

- 当前正在执行哪个研究阶段
- 当前正在调用哪个子智能体
- 当前正在调用哪个工具
- 当前是否正在检索互联网
- 当前是否正在读取网页
- 当前是否正在检索 RAGFlow
- 当前是否正在整理事实卡片
- 当前是否正在生成报告
- 当前生成了哪些中间文件或最终文件

事件格式建议：

```json
{
  "type": "agent_event",
  "event": "tool_call_started",
  "trace_id": "7f1c9c5d8b9b4d6a",
  "project_id": "project-001",
  "task_id": "task-001",
  "agent_name": "search-agent",
  "tool_name": "external_search",
  "message": "正在执行公开互联网检索",
  "data": {
    "query_summary": "新能源汽车出口趋势",
    "max_results": 5
  },
  "timestamp": "2026-06-07T12:00:00Z"
}
```

建议事件类型：

| 事件 | 含义 |
| --- | --- |
| `task_started` | 后台任务开始执行 |
| `task_progress` | 后台任务进度更新 |
| `agent_call_started` | 子智能体开始执行 |
| `agent_call_finished` | 子智能体执行完成 |
| `tool_call_started` | 工具开始调用 |
| `tool_call_finished` | 工具调用完成 |
| `artifact_created` | 生成了文件、报告或中间产物 |
| `task_finished` | 后台任务成功完成 |
| `task_failed` | 后台任务执行失败 |

这样前端不需要只靠轮询任务状态，而是可以实时展示 Agent 的执行过程。

## 14. ContextVar 会话上下文

Agent 系统中很多函数调用链路很深，如果每个工具都手动传递 `project_id`、`task_id`、`trace_id`，代码会变得很啰嗦。

建议使用 Python 的 `contextvars.ContextVar` 保存当前任务上下文。

推荐保存：

```text
trace_id
request_id
project_id
task_id
session_dir
```

示例设计：

```python
from contextvars import ContextVar

trace_id_ctx = ContextVar("trace_id", default=None)
project_id_ctx = ContextVar("project_id", default=None)
task_id_ctx = ContextVar("task_id", default=None)
session_dir_ctx = ContextVar("session_dir", default=None)
```

在后台任务开始时设置上下文：

```python
trace_token = trace_id_ctx.set(trace_id)
project_token = project_id_ctx.set(project_id)
task_token = task_id_ctx.set(task_id)
session_token = session_dir_ctx.set(session_dir)
```

在任务结束时重置上下文：

```python
trace_id_ctx.reset(trace_token)
project_id_ctx.reset(project_token)
task_id_ctx.reset(task_token)
session_dir_ctx.reset(session_token)
```

这样工具函数内部可以直接读取当前上下文：

```python
trace_id = trace_id_ctx.get()
project_id = project_id_ctx.get()
task_id = task_id_ctx.get()
```

好处：

- 避免上下文参数层层传递
- 避免并发任务之间串数据
- 方便日志自动带上 trace 信息
- 方便 WebSocket/SSE 事件自动带上任务信息
- 方便工具访问当前任务的工作目录

## 15. Agent 流式事件解析

DeepAgents / LangGraph 执行时可以使用流式输出能力，把每一步 Agent 决策过程拆出来。

建议在研究智能体调用时支持两种模式：

```text
invoke 模式：只返回最终结构化结果
stream 模式：边执行边产出事件
```

生产后端主要依赖最终结构化结果；前端展示和教学演示可以使用 stream 模式。

流式事件解析重点关注：

- `AIMessage.tool_calls`：识别模型准备调用哪个工具
- `task` 工具调用：识别模型正在委派哪个子智能体
- 最终 `content`：识别 Agent 的最终回答
- 工具返回结果：识别搜索结果数量、RAG chunk 数量、报告文件路径

建议抽象一个事件解析器：

```text
app/observability/agent_events.py
```

职责：

```text
parse_agent_stream_chunk(chunk) -> list[AgentEvent]
```

这样 DeepAgents 的底层事件结构不会直接散落在业务代码里。

## 16. 会话工作区与产物管理

研究报告类任务经常会产生中间资料和最终产物。

建议为每个研究项目或任务创建独立工作区：

```text
workspace/
  projects/
    {project_id}/
      tasks/
        {task_id}/
          inputs/
          sources/
          drafts/
          reports/
          artifacts/
```

目录职责：

| 目录 | 用途 |
| --- | --- |
| `inputs` | 用户上传文件 |
| `sources` | 网页摘要、搜索结果、RAG 检索结果 |
| `drafts` | Agent 生成的中间草稿 |
| `reports` | HTML、Markdown、PDF 报告 |
| `artifacts` | 图表、表格、其他生成文件 |

这样做的价值：

- 每个任务的文件互相隔离
- 便于追溯报告生成过程
- 便于前端展示产物列表
- 便于后续导出完整研究包
- 便于清理过期文件

生产环境中不建议长期依赖本地磁盘，可以先本地落盘，后续迁移到 MinIO 或云对象存储。

建议新增文件产物记录：

```text
research_artifacts
```

字段建议：

```text
artifact_id
project_id
task_id
artifact_type
filename
storage_path
content_type
size
created_at
```

## 17. 文件上传和文件读取工具

研究工作台可以支持用户上传内部资料，例如：

- PDF
- Word
- Excel
- Markdown
- TXT

上传文件后，Agent 可以优先读取这些资料，再结合互联网检索和 RAGFlow 检索生成报告。

建议增加：

```text
POST /api/v1/research-projects/{project_id}/files
GET  /api/v1/research-projects/{project_id}/files
GET  /api/v1/research-projects/{project_id}/artifacts
```

文件读取工具建议支持：

- `.md`
- `.txt`
- `.docx`
- `.pdf`
- `.xlsx`

工具返回内容时需要限制长度，避免一次性把大文件全部塞进 LLM 上下文。

建议策略：

| 文件类型 | 读取策略 |
| --- | --- |
| TXT / Markdown | 按最大字符数截断，必要时分页读取 |
| PDF | 按页提取文本，返回页码 |
| Word | 按段落提取 |
| Excel | 返回列名、行数、前几行、统计摘要 |

同时需要注意：

- 上传文件名必须清洗，防止路径穿越
- 文件大小需要限制
- 文件类型需要白名单
- 文件读取结果不能无限制进入 Prompt
- 敏感文件内容不能完整写入日志

## 18. 前端任务监控体验

如果增加 WebSocket 或 SSE，前端可以从简单的“等待中”升级为“可解释的执行时间线”。

页面可以展示：

```text
12:00:01 研究任务开始
12:00:03 研究管理智能体正在规划大纲
12:00:08 信息检索智能体开始检索互联网
12:00:11 external_search 返回 5 条结果
12:00:16 正在读取网页正文
12:00:22 RAGFlow 返回 12 个相关片段
12:00:35 正在整理事实卡片
12:01:10 报告生成智能体开始写作
12:01:40 HTML 报告已生成
```

这类能力对 AI 项目很重要，因为用户不只关心最终结果，也关心 AI 为什么慢、现在做到哪一步、有没有真实检索资料。

## 19. 事件推送与结构化日志的关系

建议明确区分：

```text
结构化日志：给开发和运维看
实时事件流：给用户和前端看
```

结构化日志可以更详细，包含错误堆栈、耗时、模型名、结果数量等。

实时事件流应该更克制，只展示用户能理解的信息，例如：

- 正在检索资料
- 找到 5 条公开来源
- 正在整理事实卡片
- 报告已生成

不要把完整 Prompt、完整网页正文、API 错误堆栈直接推给前端。

## 21. 面试解释口径

如果面试官问：

> 你这个项目怎么体现生产级？

可以回答：

```text
我没有把它停留在单进程 Demo。MVP 阶段使用 FastAPI 的 asyncio.create_task 跑通链路，但我也设计了生产任务后端，把生成大纲、修改大纲、生成报告这些长耗时任务迁移到 Celery Worker。API 只负责接收请求和创建任务，Worker 负责执行 Agent 和保存结果。这样 API 重启不会直接影响任务调度，也方便后续做 Worker 横向扩展、失败重试和并发控制。

另外我设计了 trace_id 链路追踪，把 HTTP 请求、MongoDB 任务记录、Celery task、Agent 调用、LLM 调用和搜索工具调用串起来。报告生成失败时，可以判断是模型超时、外部搜索失败、RAGFlow 无结果，还是报告结构化解析失败。
```

如果面试官问：

> AI 写代码很快，为什么这个项目还算两个月？

可以回答：

```text
代码生成确实借助 AI 提效很多，但这个项目主要时间不在 CRUD 代码，而在研究流程建模、Agent 职责拆分、Prompt 迭代、任务状态设计、可追溯来源设计和生产化改造方案。尤其是长耗时任务和 LLM 链路追踪，如果只是 Demo 可以不做，但要作为企业内部工具，就必须考虑任务可靠性和问题定位能力。
```

## 22. 推荐实施顺序

建议按以下顺序落地：

1. 新增日志和追踪模块，先让现有 asyncio 模式也能输出结构化 trace 日志。
2. 在 HTTP 中间件中生成 `trace_id`，并写入响应头。
3. 在创建后台任务时，把 `trace_id` 保存到任务记录中。
4. 在 Agent 调用入口记录开始、结束、耗时、状态。
5. 在 external_search、ragflow_search、web_reader、report_writer 中记录工具调用日志。
6. 增加 ContextVar 上下文，保存 trace_id、project_id、task_id 和 session_dir。
7. 增加 WebSocket 或 SSE 任务事件流，支持前端展示 Agent 执行时间线。
8. 增加 Agent 流式事件解析器，识别子智能体调用、工具调用和最终结果。
9. 增加项目工作区和文件产物记录。
10. 新增 Celery 配置和 Celery 任务文件。
11. 增加 `TASK_BACKEND` 配置，实现 `asyncio` 和 `celery` 两种任务后端切换。
12. 增加 Docker Compose，把 API、Worker、Redis、MongoDB 编排起来。
13. 补充接口测试和任务调度测试。
14. 更新 README，把启动方式、环境变量和生产化架构写清楚。

## 24. 总结

这个项目如果只看当前代码量，更像是 AI 研究报告工作台的 MVP。

如果补上日志追踪和 Celery 任务后端，它就可以更合理地表述为：

```text
具备生产化架构设计的 AI Agent 长任务系统
```

它的生产级价值不在于页面复杂，而在于：

- 长耗时 AI 任务不阻塞 API
- 任务状态可查询
- 失败可追踪
- Agent 调用链路可定位
- 检索和报告生成过程可解释
- Worker 可以横向扩展

这几个点比简单堆业务页面更适合体现 AI 应用工程能力。
